# 0011. Frontend state management

- **Status:** accepted
- **Date:** 2026-07-18
- **Context:** The frontend has two pages — Product List and Cart — that each fetch data from the REST API, show loading and error states, and (for the cart) mutate server state via add/remove. We need a state approach that keeps the UI thin, keeps fetch/orchestration logic unit-testable without React, surfaces API errors meaningfully (no silent failures, per [`Requirement.md`](../../Requirement.md) lines 59–63), and does not block a later move to MPA or SSR. We want to avoid Redux and keep things simple. Rendering stays client-side per [ADR 0001](0001-stack-and-rendering.md); the FE↔BE type boundary follows [ADR 0002](0002-type-boundary.md).
- **Decision:** A unidirectional flow per page — **View → Controller → Reducer (via `useReducer`) → state through Context → View** — with the API client and a pure DTO→view-model mapper on the side. No Redux, no external store. Each item below is a standing decision:
  - **Routing:** SPA with `react-router` now; each page is a self-contained module under `pages/` so it can later be promoted to its own Vite entry (MPA) or SSR-rendered.
  - **State container:** React Context + `useReducer`, one per page (`ProductsProvider`, `CartProvider`). State lives in the reducer; the provider binds the controller to `dispatch` and exposes `{ state, controller }` to the page subtree — **not raw `dispatch`**. The provider mounts at the page root, so each page (and each SSR request) gets its own instance — no module-level singleton.
  - **UI boundary:** the View **never calls `dispatch` or context actions directly** — it only invokes controller methods (e.g. `controller.remove(id)`). The controller is the **sole dispatcher**, keeping the View thin and all orchestration behind one testable seam.
  - **State shape (two channels):** an initial-load channel `{ status: 'idle' | 'loading' | 'success' | 'error', data, error }` for the resource, plus a separate mutation channel (e.g. per-line `pending` / `error`), so a single line action does not blank the whole page. This drives both the loading animation and the error banner.
  - **Reducer:** a pure function `(state, action) => state`; the single source of state transitions.
  - **Controller:** thunk-style async functions `(dispatch, api) => Promise<void>` with no React imports; they orchestrate `dispatch(loading)` → `api` call → `dispatch(success | error)` and own the DTO→view-model mapping via a **pure mapper function** (per [ADR 0002](0002-type-boundary.md)). State holds data only; currency/price is formatted in the View (`Intl.NumberFormat`).
  - **Re-render trade-off:** a Context value change re-renders all consumers (coarser than selector-based stores). Accepted for simplicity; the escape hatch is splitting state and dispatch into separate contexts (or memoizing) if it ever bites.
  - **Cart updates:** pessimistic — mark the affected line `pending`, call the API, then dispatch the authoritative `CartResponse` as the new state. Concurrency stays simple: disable a line's controls while `pending`, and ignore stale/out-of-order responses.
  - **Cancellation:** controllers thread an `AbortController` and skip `dispatch` after unmount.
  - **Per-page initialization/lifecycle:** each page mounts only the provider it needs and fetches on mount. The Product List page mounts `ProductsProvider` only (it shows no cart); the Cart page mounts `CartProvider` only. Providers reset on unmount and refetch on return — no cross-page state and no client-side cache.
  - **Testability:** the reducer is unit-tested as a pure function (dispatch an action, assert next state); controllers are unit-tested by passing a fake `dispatch` + fake `api`, invoking the interaction method, and asserting the exact actions dispatched (e.g. `remove(id)` dispatches `pending` then the authoritative-cart action) — both without React or network. The API client is an interface so it can be faked; the DTO→view-model mapper is a pure function tested on its own.
- **Consequences:**
  - Keeping controllers as plain `(dispatch, api)` thunks and reducers pure means the highest-value logic (orchestration + transitions) is testable without rendering — satisfying the "controller easy to unit test" goal while avoiding Redux.
  - The UI-only-calls-controller boundary keeps views thin and prevents ad-hoc `dispatch` calls scattered through components, so error/loading handling stays consistent (no silent failures).
  - Per-page providers with no singleton keep state SSR/MPA-safe: an SSR migration can add a server snapshot, and an MPA split can give each entry its own providers, without reworking the logic.
  - Resetting state on navigation means a refetch when returning to a page. Accepted for simplicity; a client cache can be added later if needed.
  - **Escape hatch:** nothing here blocks a later move to a selector-based store or a data-fetching library if the app grows.
  - **Scope:** this ADR is a decision record only; implementation is tracked in backlog P3-4a…P3-6. New dependencies (`react-router`, `@petcircle/api-types` in `apps/web`) are added at implementation time.
